
## AI 활용 방법

Claude 및 chatGPT를 활용하여 개발을 진행했습니다.
여러 AI 툴을 사용하여 교차 검증하며 사용했습니다.

- **설계 상담**: 컴포넌트 구조, 상태 관리 방식, API 연동 전략 등을 직접 설계하고 Claude/chatGPT에게 검토를 요청하는 방식으로 활용했습니다.
- **코드 리뷰**: 작성한 코드의 문제점이나 개선 방향에 대해 피드백을 받았습니다.
- **디버깅**: 발생한 문제의 원인을 스스로 분석하고 가설을 세운 뒤, Claude/chatGPT와 함께 검증하는 방식으로 해결했습니다.
- **직접 구현 요청**: 시간적 제약으로 인해 일부 반복적인 코드나 훅 분리 작업은 직접 요청하여 구현했습니다.

평소에는 문제를 먼저 정의하고 해결 방향을 제시한 뒤 검토를 받는 방식을 선호하지만, 이번 과제에서는 마감 시간의 제약으로 일부 직접 구현을 요청했습니다.

## 프로젝트 설계

### 접근 방식

API 스펙을 먼저 파악한 뒤 구현 가능한 기능과 불가능한 기능을 분류하고, 가능한 범위 내에서 요구사항을 충족하는 방향으로 개발을 진행했습니다. 기능 단위로 컴포넌트와 훅을 분리하여 역할을 명확히 했습니다.

### UI 설계

어드민 페이지의 특성을 고려해 **수정 용이성**과 **사용자 경험**을 중심으로 설계했습니다.

- **사이드 메뉴 + 콘텐츠 영역** 구조로 언제든지 메뉴 간 이동이 가능하도록 했습니다.
- 회원 목록에서 항목 클릭 시 **슬라이드 패널**이 오른쪽에서 등장해 상세 정보를 확인하고 수정할 수 있도록 했습니다. 목록을 유지하면서 상세를 볼 수 있어 어드민 업무 흐름에 적합하다고 판단했습니다.
- **회원 등록**은 단순 수정이 아닌 중요한 액션으로 판단해, 별도 전체 화면으로 분리하여 집중도를 높였습니다.

### 기술 선택

| 항목                 | 선택            | 이유                                   |
| -------------------- | --------------- | -------------------------------------- |
| 서버 상태 관리       | React Query     | 캐싱, 자동 재요청, 쿼리 키 기반 무효화 |
| 클라이언트 상태 관리 | Zustand         | 가볍고 보일러플레이트가 적음           |
| 폼 관리              | React Hook Form | 비제어 컴포넌트 기반으로 성능 우수     |
| 스타일링             | Tailwind CSS    | 빠른 UI 작성                           |

## 어려웠던 점 및 개선 방안

### API 스펙 파악

제공된 API 스펙에서 적합한 엔드포인트를 직접 파악하고 선택해야 했습니다. 응답 데이터 구조를 확인하는 데 시간이 많이 소요되었고, 특히 출결 대시보드에서 누적 출결 횟수나 출결 상태를 한 번에 조회할 수 있는 API가 없어 구현에 어려움이 있었습니다.

### 다중 API 호출로 인한 상태 관리 복잡도

하나의 화면을 구성하기 위해 회원, 세션, 출결 등 여러 API를 동시에 호출해야 했습니다. 각 응답 데이터를 조합하는 과정에서 컴포넌트 내부 상태가 복잡해졌고, 이를 관리하는 데 어려움이 있었습니다.

### 시간 제약으로 인한 코드의 미흡함

이번 과제에서 스토어를 사용하여 상태 관리를 많이 하지 못했습니다. 컴포넌트에서 useState를 호출하여 관리하여 상태 관리의 복잡성이 올라간 것 같습니다.

## 미해결 문제 및 기능

### 출결 대시보드 검색 필터

- **문제**: 기간을 기준으로 필터링 기능을 구현하지 못했습니다. 오늘 날짜를 기준으로 기간 필터를 사용하여 대시보드의 데이터값 출력 기능이 부재중입니다.
- **대안**: RHF 라이브러리를 사용하여 필터링 데이터를 가져옵니다. 대시보드에 나와있는 날짜를 필터 기준으로 재조회합니다. 이때 useMemo를 통해 원본 데이터를 업데이트 하여 반환합니다. 반환된 필터 데이터를 출력합니다. 기본값은 1년으로 잡으며, 해당 데이터가 변경될 때마다 useMemo 내부의 로직을 재계산합니다.

### 출결 대시보드 출결 현황 표시 및 회원 목록의 누적 출석 횟수 표시

- **문제**: 대시보드에서 각 회원의 누적 출결 횟수 및 현재 출결 상태를 한눈에 표시하는 것이 API 스펙상 불가능했습니다. 누적 출석 횟수를 불러오려면 회원의 id가 필요했고, 모든 회원의 id를 가지고 api 요청하는 것은 비효율적이라 생각해 누락했습니다. 출석 여부는 세션 ID 기반으로만 조회 가능한 구조라, 전체 회원의 현재 출석 상태를 단일 API로 확인할 수 없었습니다.
- **시도한 방법**: 회원별 출결 상세 API를 활용했지만, 이를 전체 회원에게 적용하려면 N번의 요청이 필요했습니다.
- **시도 가능한 대안**: 기수별 전체 출결 요약 API가 제공된다면 구현 가능합니다.
